# SALES_PLAYBOOK_GOVERNMENT_v1

**Producto:** ERP-GOB  
**Propósito:** definir una guía práctica para vender ERP-GOB a gobiernos estatales y municipales.  
**Audiencia:** dirección comercial, preventa GovTech, implantación institucional y liderazgo de producto.

---

## 1. Problema Del Mercado

Los gobiernos estatales y municipales suelen tener una arquitectura fragmentada para abastecimiento y control:
- sistema de compras por un lado;
- inventario por otro;
- patrimonial separado;
- cierre financiero desacoplado;
- auditoría basada en revisión posterior y no en control preventivo.

### 1.1 Dolor real del mercado

| Problema | Efecto institucional |
|---|---|
| Compras públicas sin expediente integral | pérdida de trazabilidad entre áreas |
| Inventario no conectado a la recepción | doble captura y diferencias operativas |
| Patrimonial aislado | activos sin control claro o sin resguardo consistente |
| Gasto sin secuencia auditable | observaciones y hallazgos de control |
| Auditoría reactiva | detección tardía de irregularidades |

### 1.2 Limitaciones de sistemas federales

Las plataformas federales o centralizadas suelen:
- cubrir solo una parte del proceso;
- enfocarse en publicación, trámite o cumplimiento mínimo;
- no resolver la operación cotidiana completa del ente;
- no integrar inventario, patrimonial y finanzas operativas de forma transversal.

### 1.3 Limitaciones de ERPs administrativos tradicionales

Los ERPs administrativos comunes suelen:
- ser fuertes en contabilidad o inventario, pero débiles en flujo contractual público;
- no incorporar checklist legal ni reglas de secuencia institucional;
- carecer de observabilidad enfocada en expediente y riesgo procedimental;
- exigir adaptación costosa para proceso gubernamental real.

### 1.4 Limitaciones de desarrollos internos

Los desarrollos internos suelen fallar en:
- gobernanza de releases;
- mantenibilidad;
- soporte;
- documentación;
- integración entre áreas;
- escalabilidad a otros entes o estados.

---

## 2. Propuesta De Valor ERP-GOB

ERP-GOB es valioso porque no resuelve una pantalla aislada.  
Resuelve el ciclo institucional completo.

### 2.1 Capacidades diferenciales
- flujo contractual completo;
- observabilidad institucional;
- control interno automatizado;
- auditoría preventiva;
- inventario operativo y patrimonial;
- wizard operativo para capturistas.

### 2.2 Beneficios concretos

| Beneficio | Resultado esperado |
|---|---|
| Reducción de corrupción e irregularidades | secuencias inválidas y riesgos detectados antes |
| Trazabilidad | expediente visible de punta a punta |
| Ahorro operativo | menos retrabajo y menos captura redundante |
| Reducción de tiempo administrativo | flujo guiado y menor dispersión entre sistemas |
| Mejora de auditoría | evidencia documental y timeline consolidado |

### 2.3 Mensaje comercial central

ERP-GOB debe venderse como:

**plataforma institucional de abastecimiento, control y trazabilidad**

No como:
- sistema de compras aislado;
- ERP genérico;
- desarrollo a medida sin producto base.

## 3. Segmentos De Cliente

### 3.1 Segmento primario
- gobiernos estatales;
- secretarías de administración;
- organismos estatales con compras, almacén e inventario;
- universidades públicas estatales.

### 3.2 Segmento secundario
- gobiernos municipales grandes;
- organismos descentralizados;
- poderes y entes autónomos con volumen administrativo relevante.

### 3.3 Segmento especializado
- instituciones de seguridad pública;
- organismos con control patrimonial intensivo;
- áreas de contraloría interna y OIC que requieren trazabilidad y observabilidad.

### 3.4 Perfil ideal de cliente

ERP-GOB encaja mejor cuando el cliente tiene:
- observaciones de auditoría recurrentes;
- procesos fragmentados;
- necesidad de conectar compras con inventario y patrimonial;
- interés en control preventivo, no solo reporte posterior.

---

## 4. Modelo De Venta

### 4.1 Modelo recomendado

La venta debe estructurarse como:
- licencia anual institucional;
- implementación inicial;
- soporte anual;
- módulos opcionales.

### 4.2 Razón del modelo

Este esquema funciona mejor en gobierno porque:
- facilita presupuesto anual;
- separa claramente producto y servicios;
- permite entrar con piloto y crecer por módulos;
- evita vender el sistema como proyecto artesanal.

### 4.3 Estructura comercial sugerida

| Componente | Naturaleza comercial |
|---|---|
| Licencia institucional | recurrente anual |
| Implantación | pago único inicial |
| Parametrización normativa | servicio especializado |
| Soporte y mantenimiento | recurrente anual |
| Capacitación | servicio complementario |

## 6. Proceso De Implantación

La implantación debe ser institucional, no improvisada.

### Fase 1. Diagnóstico institucional
- revisión del proceso actual;
- identificación de sistemas existentes;
- mapeo de actores, riesgos y normativa.

### Fase 2. Parametrización normativa
- checklist legal;
- tipos de procedimiento;
- branding;
- configuración institucional;
- módulos activos.

### Fase 3. Migración de datos
- catálogos;
- usuarios;
- proveedores;
- productos;
- inventario base;
- activos patrimoniales si aplica.

### Fase 4. Capacitación
Por rol:
- capturista;
- revisor;
- finanzas;
- OIC;
- administrador.

### Fase 5. Piloto
- uno o dos procedimientos reales;
- acompañamiento cercano;
- evidencia UAT;
- ajuste fino de operación.

### Fase 6. Operación formal
- despliegue controlado;
- soporte;
- monitoreo;
- seguimiento de adopción.

## 7. Tiempos De Implementación

Los tiempos dependen del alcance y la madurez del cliente, pero el marco de referencia comercial puede ser este:

| Escenario | Tiempo estimado |
|---|---|
| Demo funcional | 1 día |
| Parametrización demo/piloto | 1 a 2 semanas |
| Piloto institucional | 4 a 6 semanas |

### Lectura correcta

No vender tiempos irreales de “producción en una semana” si el cliente requiere:
- migración;
- capacitación;
- adaptación normativa;
- operación multiárea.

## 8. ROI Para Gobiernos

ERP-GOB debe justificarse en retorno institucional, no solo tecnológico.

### 8.1 Ahorro de tiempo administrativo
- menor captura duplicada;
- menos conciliación manual entre áreas;
- menos búsquedas de documentos y evidencias;
- menor tiempo de armado de expediente.

### 8.2 Reducción de irregularidades
- reglas de observabilidad;
- checklist legal;
- alertas de secuencia;
- visibilidad temprana de riesgos.

### 8.3 Mejora de auditoría
- timeline consolidado;
- evidencia documental vinculada;
- trazabilidad de operación;
- control interno documentado.

### 8.4 Trazabilidad del gasto
- relación entre expediente, orden, recepción, inventario y pago;
- mejor defensa institucional ante revisión.

### 8.5 Narrativa de ROI sugerida

El retorno no debe venderse solo como ahorro de licencias.  
Debe venderse como:
- menor riesgo administrativo;
- menor costo de retrabajo;
- menor tiempo de respuesta a auditoría;
- mayor control operativo.

## 9. Caso De Uso De Referencia

### Escenario de referencia

Un ente estatal necesita adquirir consumibles.

Flujo:

`expediente -> procedimiento -> orden -> recepción -> inventario -> pago -> auditoría`

### Secuencia institucional
1. Se crea expediente.
2. Se registra investigación de mercado.
3. Se completa checklist legal.
4. Se formalizan necesidades.
5. Se crea procedimiento.
6. Se revisa cuadro comparativo.
7. Se emite orden.
8. Se registra recepción.
9. Se actualiza inventario.
10. Se registra factura.
11. Se genera devengo.
12. Se registra pago.
13. OIC revisa timeline, alertas y evidencia.

### Valor comercial del caso

Este escenario demuestra:
- integración real entre módulos;
- control secuencial;
- trazabilidad de gasto;
- capacidad de auditoría preventiva.

## 10. Argumentos De Venta

### 10.1 Frente a sistemas internos

ERP-GOB es mejor porque:
- ya existe como producto funcional;
- tiene arquitectura modular;
- tiene documentación, suite reproducible y baseline;
- no depende totalmente de personas clave internas.

### 10.2 Frente a plataformas federales

ERP-GOB es mejor porque:
- no se queda en una fase parcial del proceso;
- opera el expediente completo;
- integra inventario y patrimonial;
- permite observabilidad y control a nivel institución.

### 10.3 Frente a ERPs administrativos

ERP-GOB es mejor porque:
- entiende el procedimiento público;
- integra checklist legal y observabilidad;
- conecta operación con control interno;
- está orientado a expediente, no solo a catálogo o contabilidad.

### 10.4 Mensaje comercial final

ERP-GOB no es solo un sistema para capturar compras.

Es una plataforma para:
- ejecutar abastecimiento público;
- controlar secuencia institucional;
- integrar inventario y patrimonio;
- cerrar financieramente el expediente;
- dejar evidencia auditable desde el origen.

---

## Conclusión

La venta de ERP-GOB debe plantearse como venta de una **capacidad institucional**, no solo de un software.

La conversación con el cliente debe girar sobre:
- control;
- trazabilidad;
- reducción de observaciones;
- integración operativa;
- madurez institucional.

Ese es el ángulo correcto para vender a:
- gobiernos estatales;
- municipios grandes;
- organismos autónomos;
- universidades públicas;
- instituciones con presión real de auditoría y control.
